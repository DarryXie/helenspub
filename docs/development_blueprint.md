# 鸡尾酒数据库项目开发蓝图与实施标准

## 1. 文档定位

本文档是鸡尾酒数据库项目后续代码开发的统一标准，用于指导 Codex 和人工开发协作时的实现方式、目录组织、模块拆分、变更规则和扩展方式。

本文档目标：

- 让项目具备清晰的模块边界
- 让各个子功能可以独立开发、测试和替换
- 让后续新增功能时尽量少改旧代码
- 让前端、后端、数据库和权限系统保持一致
- 作为后续开发、重构和评审的标准依据

## 2. 总体开发目标

项目一期需要同时支撑三套业务界面：

- 前台公开端：用于鸡尾酒菜单展示，直接挂载站点根路径 `/`、`/menu`，移动端优先
- 前台业务端：用于管理员和服务员登录、查看配方、管理待制作任务，统一挂载 `/app`，移动端优先
- 后台管理端：用于鸡尾酒、分类、标签、原料、图片、用户和任务管理，桌面 Web 优先

开发时必须遵守以下核心原则：

- 模块化优先
- 低耦合优先
- 单一职责优先
- 配置优先于硬编码
- 扩展优先于临时拼接
- 尽量在不破坏现有模块的情况下增加新功能

## 3. 核心架构原则

## 3.1 模块独立原则

每个业务能力必须作为独立模块设计，至少满足以下要求：

- 有独立的目录
- 有独立的数据模型
- 有独立的接口层
- 有独立的业务逻辑层
- 有独立的前端页面或组件集合

禁止将多个业务能力混写在同一个大文件中。

建议的一期核心业务模块：

- 认证与用户模块
- 角色与权限模块
- 鸡尾酒模块
- 分类模块
- 标签模块
- 原料模块
- 图片模块
- 配方模块
- 待制作任务模块
- 系统公共模块

## 3.2 分层原则

无论前端还是后端，都必须按层拆分，避免页面或接口直接承载全部逻辑。

后端至少拆分为：

- Router / Controller 层：接收请求、返回响应
- Service 层：业务逻辑
- Repository / DAO 层：数据库读写
- Model / Entity 层：数据结构定义
- Validator 层：参数校验
- Policy / Permission 层：权限判断

前端至少拆分为：

- Page 层：页面组织与路由入口
- Feature 层：业务功能单元
- Component 层：可复用组件
- Service 层：API 请求封装
- Store 层：状态管理
- Schema / Type 层：类型与数据结构

## 3.3 公共能力下沉原则

下列能力必须沉淀为公共模块，不允许在各业务模块中重复实现：

- 登录态处理
- 权限判断
- API 请求封装
- 错误提示与异常处理
- 分页处理
- 上传处理
- 表单校验
- 时间格式化
- 状态字典映射

## 3.4 开放扩展原则

开发时必须假设未来会增加以下能力：

- 客户角色
- 收藏功能
- 下单功能
- 库存功能
- 数据统计功能
- 多语言功能

因此一期实现时不得把当前业务写死为“只有管理员和服务员”“只有鸡尾酒”这样的结构。

## 4. 推荐项目结构

本文档不强制具体技术栈，但强制要求按“前端应用分离 + 后端独立服务 + 文档独立管理”的方式组织。

建议根目录结构如下：

```text
cocktail_database/
  docs/
    requirements.md
    database_design.md
    api_design.md
    page_structure.md
    development_blueprint.md

  apps/
    public-mobile/
    staff-mobile/
    admin-web/
    api-server/

  packages/
    shared-types/
    shared-utils/
    shared-constants/
    ui-components/

  database/
    schema/
    migrations/
    seeds/

  scripts/
    dev/
    build/
    deploy/
```

说明：

- `apps/` 存放可独立运行的应用
- `packages/` 存放多个应用复用的公共代码
- `database/` 存放数据库建表、迁移和初始化脚本
- `docs/` 存放所有标准文档，作为开发依据

## 5. 应用划分标准

## 5.1 `public-mobile`

职责：

- 前台公开菜单展示
- 鸡尾酒列表与详情展示
- 移动端适配
- 作为只读入口直接承载 `/`、`/menu`、`/cocktails/:id`

禁止承担：

- 后台管理逻辑
- 待制作复杂管理逻辑
- 用户管理逻辑
- 登录后点单与任务状态更新逻辑

## 5.2 `staff-mobile`

职责：

- 管理员 / 服务员登录
- 查看鸡尾酒配方
- 新增待制作任务
- 编辑待制作任务
- 更新待制作任务状态
- 承载 `/app/login` 起始的受保护业务前台入口

禁止承担：

- 主数据维护
- 分类和标签管理
- 原料管理

实现要求：

- 部署时应保证 `staff-mobile` 以 `/app` 为入口路径。
- 若开发环境仍以根路径运行，也应通过路由基座或部署映射对齐到 `/app`，避免文档与实际 URL 不一致。

## 5.3 `admin-web`

职责：

- 后台管理主数据
- 发布鸡尾酒内容
- 用户与角色管理
- 待制作任务管理总览

## 5.4 `api-server`

职责：

- 提供全部 API 接口
- 统一处理鉴权、权限、数据校验和业务逻辑
- 统一访问数据库

禁止在前端应用中直接拼 SQL 或绕过后端处理核心业务规则。

## 6. 后端开发蓝图

## 6.1 后端目录标准

建议 `api-server` 目录结构如下：

```text
api-server/
  src/
    app/
    config/
    common/
    modules/
      auth/
      users/
      roles/
      cocktails/
      categories/
      tags/
      ingredients/
      images/
      recipes/
      production-tasks/
    database/
    types/
```

## 6.2 模块内部标准结构

每个业务模块建议采用统一结构：

```text
modules/
  cocktails/
    controller/
    service/
    repository/
    dto/
    validator/
    mapper/
    types/
    index.ts
```

如果项目后续复杂度增加，可进一步演进为：

```text
modules/
  cocktails/
    controllers/
    services/
    repositories/
    entities/
    dto/
    policies/
    validators/
    constants/
    tests/
```

## 6.3 后端模块职责边界

### `auth`

负责：

- 登录
- token 签发
- 当前用户信息
- 退出登录

不负责：

- 用户资料编辑
- 用户角色配置

### `users`

负责：

- 用户 CRUD
- 用户状态管理
- 用户基础资料

### `roles`

负责：

- 角色读取
- 角色配置扩展
- 权限基础映射

### `cocktails`

负责：

- 鸡尾酒基础资料
- 发布状态
- 前台可见性

不直接负责：

- 配方明细保存细节
- 图片上传物理处理

### `recipes`

负责：

- 配方明细读写
- 原料顺序
- 用量与单位校验

### `images`

负责：

- 上传元数据管理
- 图片关联管理

### `production-tasks`

负责：

- 待制作任务 CRUD
- 状态流转
- 任务日志
- 任务查询与筛选

## 6.4 Service 设计标准

Service 必须只处理业务逻辑，不直接关心 HTTP 表现层细节。

Service 应负责：

- 业务规则判断
- 跨表事务控制
- 状态流转校验
- 组合多个 Repository 数据

Service 不应负责：

- 直接组装 Express / Nest 原始响应
- 页面 UI 字段格式拼接

## 6.5 Repository 设计标准

Repository 只负责数据访问。

Repository 应负责：

- 查询条件构建
- 单表或明确的关联表读写
- 返回明确结构

Repository 不应负责：

- 权限判断
- 状态机校验
- 业务提示文案

## 6.6 DTO 与 Validator 标准

所有写接口必须有明确 DTO 和参数校验规则。

例如：

- `CreateCocktailDto`
- `UpdateCocktailDto`
- `CreateProductionTaskDto`
- `UpdateProductionTaskStatusDto`

校验内容至少包括：

- 类型
- 必填
- 长度
- 枚举值
- 数量范围

## 6.7 权限层标准

权限不能散落在各 Controller 中硬编码。

必须采用统一策略：

- 路由声明需要什么角色
- Policy / Guard 统一判断
- 模块内部按角色细化规则时，放在 Policy 层

例如：

- `admin` 可访问 `/admin/*`
- `staff` 和 `admin` 可访问 `/app/*`
- `guest` 可直接访问 `/`、`/menu`
- `customer` 一期仅保留数据结构，不提供独立业务入口

## 6.8 事务标准

以下操作必须使用事务：

- 新增鸡尾酒并保存分类、标签、配方、图片关系
- 编辑鸡尾酒并同步更新配方、标签、分类
- 创建待制作任务并写入任务日志
- 更新待制作任务状态并写入日志

## 7. 前端开发蓝图

## 7.1 三端独立原则

前台公开端、前台业务端、后台管理端必须视为三个独立前端应用。

原因：

- 页面目标不同
- 权限不同
- 交互形态不同
- 响应式要求不同

禁止在同一个前端应用中通过大量 `if role === ...` 强行拼接全部页面逻辑。

补充约定：

- 公开菜单入口只负责只读浏览，不承载登录后的点单和任务操作。
- 业务前台入口只负责登录、查配方、创建待制作任务和更新任务状态。
- 不通过在同一个前端应用里切换角色菜单来同时承载公开端和业务端。

## 7.2 前端目录标准

每个前端应用建议结构如下：

```text
src/
  app/
  pages/
  features/
  components/
  services/
  store/
  hooks/
  utils/
  constants/
  types/
  styles/
```

## 7.3 页面与业务功能拆分原则

页面只负责布局、组合和路由。

真正的业务能力要放在 `features/` 中，例如：

```text
features/
  auth/
  cocktail-list/
  cocktail-detail/
  production-task-list/
  production-task-form/
  recipe-viewer/
```

这样做的目的：

- 页面更薄
- 功能更容易迁移
- 新页面可以复用已有 feature

## 7.4 组件分层标准

组件至少分为三类：

- 通用 UI 组件
- 业务组件
- 页面骨架组件

示例：

- 通用 UI：按钮、输入框、弹窗、表格、标签、分页
- 业务组件：鸡尾酒卡片、配方列表、任务状态标签
- 页面骨架：移动端顶部栏、后台侧边栏、筛选工具栏

## 7.5 前端状态管理标准

状态分为三类：

- 服务端数据状态
- 页面临时状态
- 全局会话状态

建议规则：

- 列表和详情数据走数据请求层管理
- 登录用户信息走全局会话状态
- 弹窗开关、表单输入等走局部状态

禁止把所有状态都塞进一个巨大的全局 store。

## 7.6 API 请求封装标准

前端不得在页面中直接写裸请求。

必须统一封装在 `services/` 中，例如：

```text
services/
  auth.service.ts
  cocktails.service.ts
  productionTasks.service.ts
  users.service.ts
```

请求封装必须统一处理：

- baseURL
- token 注入
- 错误码处理
- 超时
- 上传格式

## 7.7 类型共享标准

涉及跨端复用的数据结构，应优先沉淀到 `packages/shared-types/`。

例如：

- `UserSummary`
- `CocktailItem`
- `CocktailDetail`
- `ProductionTask`
- `Pagination`

禁止三个前端和一个后端各自手写一套字段定义。

## 7.8 路由标准

路由按业务分区，不按文件数量分区。

示例：

- 公开端：`/`、`/menu`、`/cocktails/:id`
- 业务端：`/app/login`、`/app/tasks`、`/app/cocktails`
- 后台端：`/admin/cocktails`、`/admin/users`

必须支持：

- 未登录拦截
- 无权限重定向
- 登录后自动跳转默认首页

术语约定：

- “服务员点单”在一期实现中统一指“创建待制作任务”。
- API、数据库和代码层继续使用 `production-task` / `待制作任务` 命名，不引入新的 `order` 主模型。

## 8. 数据库与数据访问标准

## 8.1 Schema 变更原则

任何数据库字段变更必须满足：

- 先更新文档
- 再新增 migration
- 再修改代码

禁止直接在线上结构上手工修改后再补代码。

## 8.2 Migration 标准

数据库变更必须通过 migration 管理。

建议命名方式：

- `20260617_init_schema.sql`
- `20260618_add_task_priority.sql`

每次 migration 只做一类明确变更，避免大杂烩。

## 8.3 Seed 标准

初始化数据应独立管理：

- 角色
- 基础分类
- 基础标签
- 基础原料
- 默认管理员账号

## 8.4 快照字段标准

以下场景必须保留快照字段，避免历史数据被主数据修改影响：

- 待制作任务中的鸡尾酒名称
- 配方中的原料名称

后续如果增加订单，也应保留价格和名称快照。

## 9. 业务规则实现标准

## 9.1 鸡尾酒发布规则

- 鸡尾酒至少有一个分类
- 鸡尾酒至少有一条配方才允许发布
- 隐藏或未发布的鸡尾酒不能出现在公开前台

这些规则必须写在 Service 层，不允许只依赖前端限制。

## 9.2 待制作任务状态规则

一期状态：

- `pending`
- `in_progress`
- `completed`

状态流转必须受控：

- `pending -> in_progress`
- `pending -> completed`
- `in_progress -> completed`

默认不允许：

- `completed -> pending`
- `completed -> in_progress`

如果未来需要回退，必须新增明确业务规则而不是临时放开。

## 9.3 用户角色规则

- `admin` 可以访问后台和业务前台
- `staff` 只能访问业务前台
- `guest` 可直接访问公开菜单入口
- `customer` 一期只保留数据结构，不开放业务页面或独立入口

角色判断必须基于角色编码，不基于前端页面名称或按钮名称。

## 10. 新增功能接入标准

后续新增功能时，必须按以下流程实施：

1. 先补需求文档或变更说明
2. 判断是否新增独立模块
3. 更新数据库设计和 migration
4. 新增后端模块或扩展既有模块
5. 新增前端 feature，而不是把逻辑塞进旧页面
6. 补充接口文档
7. 补充测试

### 10.1 什么时候新增独立模块

满足以下任一条件时，优先新增模块而不是塞进旧模块：

- 有独立数据表
- 有独立页面入口
- 有独立权限逻辑
- 有独立业务状态流转

例如以下功能应独立成模块：

- 收藏功能
- 订单功能
- 库存功能
- 消息通知功能

## 11. 命名与代码规范

## 11.1 命名规范

- 数据库表名：复数英文下划线
- 字段名：小写下划线
- TypeScript 类型名：大驼峰
- 变量与函数：小驼峰
- 组件名：大驼峰
- 文件名建议与职责对应

示例：

- `production_tasks`
- `CreateProductionTaskDto`
- `productionTaskService`
- `ProductionTaskListPage`

## 11.2 文件职责规范

一个文件只做一件清晰的事。

禁止出现：

- 一个页面文件同时写请求、表单校验、复杂状态机和 UI 组件定义
- 一个 service 文件同时包含多个完全无关模块

## 11.3 常量规范

状态值、角色值、分页默认值、上传限制等必须提取为常量。

例如：

- `ROLE_CODES`
- `TASK_STATUS`
- `COCKTAIL_PUBLISH_STATUS`

## 11.4 配置规范

以下内容必须走配置，不允许硬编码：

- API 地址
- 上传路径
- token 过期时间
- 分页默认值
- 文件大小限制

## 12. 测试与质量标准

## 12.1 后端测试重点

必须优先测试以下内容：

- 登录与权限控制
- 鸡尾酒发布规则
- 待制作任务状态流转
- 配方保存逻辑
- 删除受引用数据时的限制

## 12.2 前端测试重点

必须优先测试以下内容：

- 登录拦截
- 列表加载与筛选
- 表单校验
- 任务状态更新
- 不同角色下的菜单显示

## 12.3 回归标准

每新增一个模块或修改一个核心模块，至少检查：

- 接口是否兼容
- 权限是否受影响
- 路由是否受影响
- 数据库迁移是否可执行
- 旧页面是否被破坏

## 13. 文档联动标准

后续开发中，如果发生以下变更，必须同步更新文档：

- 新增数据表或字段
- 新增接口
- 新增角色
- 新增页面
- 变更状态流转

应同步更新的文档：

- `requirements.md`
- `database_design.md`
- `api_design.md`
- `page_structure.md`
- `development_blueprint.md`

## 14. Codex 开发执行规则

后续让 Codex 参与开发时，默认应遵守以下执行规则：

- 先阅读 `docs/` 下标准文档，再开始写代码
- 优先新增模块，不直接污染旧模块
- 优先复用公共包，不复制粘贴代码
- 修改数据库时必须同时新增 migration
- 新接口必须同步补文档
- 新页面必须遵守既定路由分区
- 新增角色相关逻辑时必须走统一权限层

如果 Codex 遇到需要实现新功能，应优先输出：

1. 涉及模块清单
2. 数据结构是否变化
3. 接口变更点
4. 页面变更点
5. 实施顺序

## 15. 一期推荐实施顺序

建议后续按以下顺序开发：
